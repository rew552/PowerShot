# PowerShot 技術仕様書 (ARCHITECTURE)

PowerShot v3.2 の内部設計、動作原理、および技術的な特徴について解説します。

---

## 1. 全体アーキテクチャ

PowerShot は、**PowerShell による柔軟なスクリプト実行**と、**C# (WPF) による高度な UI/システム操作**を組み合わせたハイブリッド構成を採用しています。

### 1.1 ハイブリッド実行モデル
- **起動スクリプト (PowerShell)**: 環境のセットアップ、DPI設定、C# ファイルの読み込み、および `Add-Type` による動的コンパイルを担当します。
- **コアロジック (C#)**: クリップボード監視、ホットキー制御、画像処理（GDI+）、およびメイン UI (WPF) を担当します。

この構成により、ビルド済みのバイナリ（EXE）を配布することなく、ソースコードのまま配布・実行が可能になっています。

---

## 2. クリップボード監視メカニズム

従来のポーリング方式（一定間隔で確認）ではなく、Windows のメッセージベースのイベント駆動方式を採用しています。

### 2.1 WM_CLIPBOARDUPDATE
- `user32.dll` の `AddClipboardFormatListener` を呼び出し、OS にクリップボードの更新を通知するよう登録します。
- 非表示のウィンドウ（Message-Only Window）で `WM_CLIPBOARDUPDATE (0x031D)` メッセージを待ち受けます。
- **メリット**: CPU 負荷がほぼゼロであり、クリップボードが更新された瞬間に即座に反応できます。

---

## 3. 独自ホットキー (Shift + PrintScreen)

Windows 標準のキャプチャ機能とは別に、独自のアクティブモニターキャプチャを実現しています。

- `RegisterHotKey` API を使用して、システム全体で有効なホットキーを登録します。
- ホットキーが押されると、マウスカーソルの位置から「現在アクティブなモニター」を特定し、その範囲を `Graphics.CopyFromScreen` でキャプチャします。
- キャプチャした画像は一度クリップボードへ送出され、前述の「クリップボード監視」によって自動的に保存 UI が起動する流れになっています。

---

## 4. UI 構成 (WPF / XAML)

UI は完全に分離された XAML ファイルによって定義されています。

- **動的ロード**: `XamlReader.Load` を使用して、実行時に `MainWindow.xaml` を読み込みます。これにより、UI の微調整を C# の再コンパイルなしで行うことが可能です。
- **STA スレッド**: クリップボード操作や WPF UI の実行には STA (Single Threaded Apartment) が必要であるため、`powershot.bat` 内で `powershell -sta` として起動を強制しています。

---

## 5. 独立型ビューア (PowerShot Viewer)

過去のキャプチャ履歴を閲覧するビューアは、ウェブ技術をローカルで活用する設計になっています。

### 5.1 オフライン・サーバーレス設計
- ローカル環境でのブラウザのセキュリティ制限（CORS）を回避するため、C# 側でディレクトリをスキャンして得られたデータ（JSON）を、HTML テンプレート（`viewer_template.html`）の文字列として直接置換して焼き込みます。
- 生成された一時的な `.powershot_viewer.html` を開くことで、ウェブサーバーを介さずに、高速かつリッチなブラウジング体験を実現しています。

---

## 6. データ永続化

- **AppSettings**: JSON 形式（`DataContractJsonSerializer`）で `src/settings.json` に保存されます。
- **SessionState**: アプリケーション実行中の状態（最後に開いたフォルダ、連番の現在値など）は、メモリ上で保持され、必要に応じて UI 間で共有されます。
